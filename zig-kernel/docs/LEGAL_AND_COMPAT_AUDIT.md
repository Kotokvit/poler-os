## 1. API-совместимость: Законна ли реализация Windows API?

### 1.1. Фундаментальный принцип: Интерфейс не охраноспособен

Центральный столп всего юридического анализа — **принцип разделения интерфейса и реализации**, закреплённый в прецедентном праве США уже более 150 лет (начиная с *Baker v. Selden*, 101 U.S. 99 (1880)). API — это интерфейс. Закон охраняет выражение (implementation), но не идею, процедуру, процесс, систему или метод операции (17 U.S.C. § 102(b)).

### 1.2. Oracle v. Google, 593 U.S. 1 (2021) — Разбор в контексте POLER-OS

**Фабула:** Google скопировал ≈11 500 строк из 37 пакетов Java SE API (структура, последовательность, организация — SSO), реализовав их в Android. Oracle подала иск о нарушении авторских прав.

**Решение Верховного суда США (6-2):**

1. **API SSO является «методом операции» в смысле § 102(b)**. Суд установил, что декларации API (имена, сигнатуры, иерархия пакетов) — это функциональная необходимость для взаимодействия с экосистемой. Без копирования этих элементов невозможно обеспечить совместимость.

2. **Fair use (добросовестное использование)**. Суд применил четырёхфакторный тест и нашёл, что использование Google было добросовестным:
   - **Цель и характер**: трансформативное использование — Android — новая платформа, не замена Java SE
   - **Природа произведения**: функциональная, а не творческая
   - **Объём**: 0.4% от всего Java SE, только необходимое для совместимости
   - **Рыночный эффект**: Android не замещает Java SE на рынке десктопов

**Прямое применение к POLER-OS:**

| Критерий Oracle v. Google | Применение к POLER-OS |
|--------------------------|----------------------|
| API SSO = метод операции | NT API (NtCreateFile, NtReadFile, структура EPROCESS и т.д.) — это метод операции, необходимый для совместимости |
| Копирование только необходимого | POLER-OS реализует только те элементы NT API, которые нужны для корректной работы приложений — не более |
| Трансформативное использование | POLER-OS — не замена Windows, а новая платформа с собственной архитектурой ядра |
| Рыночный эффект | POLER-OS не конкурирует с Windows на рынке десктопных ОС; это нишевая платформа |

**Критические оговорки:**

- Решение Oracle v. Google было по fair use, а не по вопросу охраноспособности как таковой. Суд *предположил* (assumed without deciding), что API SSO вообще охраноспособен авторским правом, но решил дело на основании fair use. Это означает, что **в будущем суд может иначе оценить fair use** в другом контексте.
- POLER-OS должен избегать **ненужного копирования** — только та часть API, которая функционально необходима для совместимости. Копировать «на всякий случай» — увеличивает риск.
- **Решение не даёт carte blanche** на копирование любой документации, заголовочных файлов, или реализации.

### 1.3. Директива ЕС 2009/24/EC — Статья 6(1)

**Текст:** «Компьютерная программа, являющаяся объектом авторского права, может быть декомпилирована, если это необходимо для получения информации, необходимой для достижения совместимости независимо созданной компьютерной программы с другими программами, при соблюдении следующих условий: [...]»

**Условия:**
1. Декомпиляция выполняется лицензированным пользователем или уполномоченным лицом
2. Необходимая информация для совместимости ранее не была легко и быстро доступна
3. Декомпиляция ограничивается теми частями программы, которые необходимы для совместимости

**Значение для POLER-OS:**

- ЕС прямо признаёт **право на реверс-инжиниринг** для достижения совместимости
- Это сильная юридическая позиция для любого разработчика в юрисдикции ЕС
- Однако: это не отменяет авторского права на реализацию — можно изучать поведение, но не копировать код
- **Полезно как дополнительный аргумент**, но не как единственная опора — POLER-OS должен строить защиту на нескольких правовых основаниях

### 1.4. Lotus Development Corp. v. Borland International, 49 F.3d 807 (1st Cir. 1995)

**Фабула:** Borland реализовал меню и команду Lotus 1-2-3 в своём продукте Quattro Pro, чтобы пользователи могли использовать привычные макросы.

**Решение (подтверждено Верховным судом США равным разделением голосов 4-4):**

- Меню-команда — это **метод операции** в смысле § 102(b)
- Пользователь использует меню для управления программой — это не выражение, а функциональный элемент
- Следовательно, меню-команда **не охраноспособна** авторским правом

**Применение к POLER-OS:**

- NT API — это «меню» операционной системы для приложений. Приложения «нажимают кнопки» NtCreateFile, NtReadFile и т.д.
- Если меню Lotus 1-2-3 не охраноспособно, то имена и сигнатуры функций NT API — тем более
- **Lotus v. Borland поддерживает позицию, что API как метод операции не охраноспособен**

### 1.5. Sega v. Accolade, 977 F.2d 1510 (9th Cir. 1992)

**Фабула:** Accolade декомпилировала программы для Sega Genesis для получения информации о интерфейсе консоли, чтобы создавать совместимые игры.

**Решение:**

- Промежуточное копирование (декомпиляция) коммерческого программного обеспечения является **добросовестным использованием**, если:
  1. Цель — получение неохраноспособных элементов (интерфейсов)
  2. Нет разумного альтернативного способа получения информации
  3. Копируется только необходимое

**Значение для POLER-OS:** Это прецедент, подтверждающий право на реверс-инжиниринг для совместимости. Но POLER-OS должен использовать **чистую комнату** и не полагаться исключительно на этот прецедент — см. раздел 2.

### 1.6. Sony v. Connectix, 203 F.3d 596 (9th Cir. 2000)

**Фабула:** Connectix реверс-инжинирировала BIOS Sony PlayStation для создания эмулятора Virtual Game Station.

**Решение:**

- Промежуточное копирование BIOS является добросовестным использованием
- Эмулятор как таковой не нарушает авторских прав, если финальный продукт не содержит скопированного кода
- **Эмуляция коммерческого продукта законна**

**Прямая аналогия:** POLER-OS — это «эмулятор» Windows API (хотя и не эмулятор в традиционном смысле — это нативная реализация). Connectix поддерживает законность такого подхода.

### 1.7. Итоговый вывод по API-совместимости

**Реализация Windows NT API от третьего лица законна** при соблюдении следующих условий:

1. ✅ Не копируется код реализации Microsoft
2. ✅ Копируются только функционально необходимые элементы API (имена, сигнатуры, семантика)
3. ✅ Реализация создаётся самостоятельно (clean-room или эквивалентный процесс)
4. ✅ Не копируется документация Microsoft verbatim
5. ✅ Не используются запатентованные алгоритмы (см. раздел 5)

## 2. Чистая комната (Clean-Room Development)

### 2.1. Что такое чистая комната?

Чистая комната — это методология разработки, при которой **команда спецификаций** (чисто юридически «грязная») изучает оригинальный продукт и составляет спецификацию, а **команда реализации** (юридически «чистая») никогда не видит оригинальный код и реализует только по спецификации.

Классический пример: **PC BIOS** (Compaq, 1982). Compaq создала клон IBM PC, реализовав BIOS через чистую комнату:
- Команда A реверс-инжиниринга изучала IBM BIOS и написала спецификацию
- Команда B (изолированная) реализовала BIOS по спецификации
- Результат: юридически безупречный клон, IBM не смогла подать иск

### 2.2. Юридические требования к чистой комнате

**Минимальные требования (установленные прецедентным правом):**

1. **Физическая изоляция команд** — члены команды реализации не должны иметь доступа к оригинальному коду
2. **Документирование процесса** — каждый шаг должен быть задокументирован:
   - Кто изучал оригинальный продукт
   - Что именно было изучено
   - Какая спецификация была создана
   - Кто реализовывал по спецификации
   - Как подтверждалась изоляция
3. **Однонаправленный поток информации** — от грязной команды к чистой, никогда обратно
4. **Спецификация не должна содержать код** — только поведенческое описание (input/output, побочные эффекты, инварианты)

### 2.3. Может ли один человек делать чистую комнату?

**Технически — да, но с большими оговорками.**

Юридически чистая комната одним человеком — это **ослабленная позиция**. Причина: суд будет скептически относиться к утверждению «я забыл код, который видел вчера». Однако:

- **Wine** — самый известный проект совместимости с Windows API — исторически не использовал строгую чистую комнату. Wine полагается на:
  1. Публичную документацию (MSDN, Windows SDK headers, WDK headers)
  2. Наблюдение за поведением (black-box testing)
  3. Запрет на чтение дизассемблированного кода Microsoft

- **ReactOS** — использует чистую комнату с разделением команд (specification → implementation)

**Рекомендация для POLER-OS:**

POLER-OS может использовать **модифицированную чистую комнату** для одного человека:

1. **Фаза изучения (Dirty Phase):**
   - Изучение публичной документации (MSDN, Windows SDK, WDK)
   - Чёрный ящик: запуск Windows, наблюдение за поведением API через отладчики
   - Написание спецификации: «Функция NtCreateFile принимает параметры X, Y, Z и возвращает результат R при условиях A, B, C»
   - **НЕ дизассемблировать** код ntdll.dll, ntoskrnl.exe

2. **Фаза реализации (Clean Phase):**
   - Использовать ТОЛЬКО спецификацию, написанную на предыдущей фазе
   - Не обращаться к оригинальному коду Microsoft
   - Документировать все источники информации

3. **Документирование:**
   - Вести журнал: дата, источник, что изучено, что специфицировано
   - Git-коммиты с пометками: `[SPEC]` для спецификаций, `[IMPL]` для реализаций
   - Хранить спецификации отдельно от кода реализации

### 2.4. Публичные источники — юридически безопасные

Следующие источники **не требуют чистой комнаты**, поскольку они являются публично доступными и не нарушают авторских прав Microsoft:

| Источник | Юридический статус | Использование |
|----------|-------------------|---------------|
| Windows SDK headers (windows.h, ntdef.h и т.д.) | Публично доступные, EULA разрешает использование для разработки | ✅ Можно использовать для понимания сигнатур API |
| WDK headers (wdm.h, ntddk.h) | Публично доступные через Windows Driver Kit | ✅ Можно использовать для понимания kernel API |
| MSDN documentation | Публично доступная | ✅ Можно использовать для понимания поведения API |
| Windows Internals (Russinovich) | Опубликованная книга | ✅ Можно использовать для понимания архитектуры |
| Референс- implementation Wine | LGPL-лицензия | ⚠️ Можно изучать поведение, но НЕ копировать код (LGPL) |
| ReactOS | GPL-лицензия | ⚠️ Можно изучать поведение, но НЕ копировать код (GPL) |
| Дизассемблированный код Microsoft | ❌ Охраняемый авторским правом | ❌ НЕ использовать в чистой комнате |
| Leak-исходники Windows | ❌ Незаконные | ❌ АБСОЛЮТНО НЕ использовать |

### 2.5. Специфика чистой комнаты для POLER-OS

Учитывая текущую архитектуру POLER-OS (shim в user-mode), чистая комната применима к:

1. **poler-ntdll.dll** — реализация NT API в user-mode shim
2. **poler-kernel32.dll** — реализация Win32 API в user-mode shim
3. **PE Loader** — загрузчик PE/COFF бинарников
4. **Object Manager emulation** — если реализуется в user-mode
5. **Registry emulation** — виртуальный реестр

Для **ядра POLER-OS** чистая комрата не требуется, потому что ядро реализует собственный ABI — не NT API.

## 3. Реализация поведения ОС: EPROCESS, Object Manager и др.

### 3.1. Принцип: Поведение не охраноспособно

**EPROCESS** — это структура данных в ядре Windows, представляющая процесс. Включает поля вроде:
- `UniqueProcessId` — PID процесса
- `ActiveThreads` — количество активных потоков
- `Token` — маркер безопасности
- `VadRoot` — корень VAD-дерева
- `ObjectTable` — таблица дескрипторов

**Вопрос:** Может ли POLER-OS реализовать структуру с такими же полями и такой же семантикой?

**Ответ:** **Да**, при условии, что:
1. Код реализации не скопирован из Windows
2. Расположение полей (offset) может отличаться — POLER-OS не обязан бинарно-совместим на уровне структур
3. Имена полей — функционально обусловлены, не охраноспособны

### 3.2. Разница между поведением и кодом

| Аспект | Охраноспособен? | Пояснение |
|--------|----------------|-----------|
| Семантика NtCreateFile (создаёт файл, возвращает HANDLE) | ❌ Нет | Это поведение, метод операции |
| Код реализации NtCreateFile в ntoskrnl.exe | ✅ Да | Это конкретное выражение |
| Структура EPROCESS с полями PID, Token и т.д. | ❌ Нет (имена и семантика) | Функционально обусловлено |
| Конкретное расположение полей EPROCESS в памяти | ⚠️ Серая зона | Если скопировано — может быть доказательством копирования; если выведено логически — нет |
| Алгоритм балансировки VAD-дерева | ⚠️ Зависит | Если это стандартный AVL/red-black — не охраноспособен; если уникальный — может быть |
| Формат бинарника PE/COFF | ❌ Нет | Публичный стандарт (Microsoft PE and COFF Specification) |

### 3.3. Object Manager — специфический анализ

NT Object Manager реализует:
- Иерархическое пространство имён (`\Device`, `\??`, `\BaseNamedObjects`)
- Символические ссылки (`\??\C:` → `\Device\HarddiskVolume1`)
- Счётчик ссылок на объекты
- Дескрипторную таблицу (handle table) per-process
- Диспетчеризацию по типу объекта

**Юридический анализ:**

- **Пространство имён** — функционально обусловлено, не охраноспособно. Аналог директорий в файловой системе
- **Символические ссылки** — общепринятый концепт (UNIX symlinks, Windows reparse points)
- **Счётчик ссылок** — базовый механизм управления памятью, не охраноспособен
- **Handle table** — концептуально эквивалентен файловым дескрипторам UNIX, не охраноспособен
- **Диспетчеризация по типу** — стандартный паттерн ООП, не охраноспособен

**Вывод:** Реализация Object Manager с аналогичной семантикой полностью законна.

### 3.4. Реализация системных вызовов

NT syscall интерфейс (NtCreateFile, NtReadFile, etc.) — это API, и реализация этого API от третьего лица покрывается теми же прецедентами (Oracle v. Google, Lotus v. Borland).

**Важный нюанс — номера системных вызовов:**

Номера syscall в Windows меняются между версиями:
- Windows 10 1507: NtCreateFile = 0x55
- Windows 10 2004: NtCreateFile = 0x57
- Windows 11 22H2: NtCreateFile = 0x5A

POLER-OS не обязан и **не должен** использовать те же номера syscall — это не часть API, это внутренняя деталь реализации Windows. Приложения вызывают функции через ntdll.dll, а не напрямую через `syscall`. Номера syscall — это implementation detail, и копировать их нет нужды (и даже вредно — они нестабильны).

## 4. Обратная совместимость и запатентованные алгоритмы

### 4.1. Обратная совместимость — законна

Реализация обратной совместимости — это именно то, что прецедентное право защищает. *Sega v. Accolade* и *Sony v. Connectix* прямо говорят о праве на создание совместимых продуктов.

### 4.2. NTFS — критическая проблема патентов

**Файловая система NTFS содержит запатентованные технологии Microsoft:**

| Патент | Описание | Статус |
|--------|----------|--------|
| US 5,758,352 | Method and system for managing file system data | Истёк (1998 + 20 = 2018) |
| US 6,138,143 | File system with on-disk structure for directory | Истёк (2000 + 20 = 2020) |
| US 7,058,718 | Data processing system and method for recovering file system | Истёк (2003 + 20 = 2023) |
| US 8,185,818 | Resilient file system (ReFS) | Действующий! |

**Ключевой вывод:** Большинство патентов на NTFS **истекли**. NTFS существует с Windows NT 3.1 (1993), и патенты 1990-х уже в публичном домене.

**Однако:**
- Microsoft может иметь патенты на новые фичи NTFS (TxF, Deduplication)
- ReFS — отдельная файловая система с активными патентами
- Microsoft может подать патентный иск даже на истёкшие патенты — стоимость защиты высока

**Рекомендация для POLER-OS:**

1. **Не реализовывать NTFS** — использовать собственную ФС (POLER-FS) или открытые (ext2/4, btrfs)
2. Если NTFS необходима для совместимости — реализовать **только чтение** (mount as read-only)
3. Использовать существующие open-source реализации NTFS (ntfs-3g, Linux NTFS driver) как справочный материал — они уже прошли юридическую проверку
4. **NTFS-3G** (Tuxera) распространяется под GPL — можно изучать поведение, нельзя копировать код (если POLER-OS не GPL)

### 4.3. Другие запатентованные технологии в Windows

| Технология | Патентный риск | Рекомендация |
|-----------|----------------|--------------|
| NTFS | Средний (патенты истекают, но новые фичи — нет) | Не реализовывать или read-only |
| ReFS | Высокий | Не реализовывать |
| BitLocker | Высокий | Не реализовывать |
| SMB2/3 | Средний (некоторые расширения запатентованы) | Реализовать базовый SMB2 |
| Direct3D | Низкий (Vulkan/DXVK — открытые реализации) | Использовать DXVK/Vulkan |
| WinSock | Низкий | Реализовать по RFC |
| Print Spooler | Низкий | Упрощённая реализация |

## 5. IP-риски: Детальный анализ

### 5.1. Авторское право (Copyright)

**Что охраняется:**
- Исходный код Windows (ntoskrnl.exe, ntdll.dll и т.д.)
- Документация Microsoft (MSDN, Windows SDK docs)
- Заголовочные файлы Windows SDK (с оговорками — см. ниже)
- Иконки, шрифты, графические ресурсы Windows

**Что НЕ охраняется:**
- Имена функций (NtCreateFile, CreateFileW)
- Сигнатуры функций (параметры, типы возвращаемых значений)
- Семантика поведения (функция X создаёт файл и возвращает дескриптор)
- Структуры данных (EPROCESS, ETHREAD) — имена и семантика полей
- Протоколы (ALPC, RPC) — если они публично документированы

**Реальный риск:** Низкий, если POLER-OS не копирует код. Основной риск — случайное копирование фрагментов кода при реверс-инжиниринге.

**Митигация:**
- Чистая комната (даже модифицированная)
- Использование публичных спецификаций
- Автоматическая проверка на совпадение с дизассемблированным кодом Microsoft

### 5.2. Патентное право (Patent)

**Что охраняется:**
- Конкретные запатентованные алгоритмы и методы
- См. раздел 4.2 (NTFS патенты)

**Что НЕ охраняется:**
- Общие концепции ОС (планирование, виртуальная память, IPC)
- Стандартные структуры данных (AVL-деревья, хеш-таблицы)
- Общеизвестные алгоритмы

**Реальный риск:** Средний. Microsoft имеет обширный патентный портфель. Однако:
- Microsoft исторически **не подаёт патентные иски против open-source проектов совместимости** (Wine, ReactOS, Samba)
- Патентные иски — дорого и долго (минимум $2-5M до суда)
- Microsoft больше не доминирует на рынке ОС — судебный иск против маленького проекта будет выглядеть как буллинг

**Митигация:**
- Не реализовывать запатентованные технологии (NTFS, ReFS, BitLocker)
- Патентный поиск перед реализацией спорных компонентов
- Рассмотреть участие в OIN (Open Invention Network) для патентной защиты

### 5.3. Товарные знаки (Trademark)

**Что охраняется:**
- «Windows» — зарегистрированный товарный знак Microsoft
- «Microsoft» — зарегистрированный товарный знак
- Логотип Windows
- «NT», «Win32» — могут быть охраняемы

**Что POLER-OS НЕ должен делать:**
- Называть продукт «Windows», «Windows-compatible», «Microsoft Windows»
- Использовать логотип Windows
- Утверждать «это Windows» или «сертифицировано Microsoft»
- Использовать товарные знаки в доменных именах без разрешения

**Что POLER-OS МОЖЕТ делать:**
- Говорить «совместим с Windows API» (nominative fair use)
- Использовать технические термины (NT API, Win32 API) для описания совместимости
- Реализовывать API с теми же именами функций — это функциональное использование, не товарный знак

**Реальный риск:** Низкий. Товарные знаки — самая простая для избежания категория IP.

### 5.4. Коммерческая тайна (Trade Secret)

**Что охраняется:**
- Недокументированные внутренние алгоритмы Windows
- Исходный код Windows (если не утёк)
- Внутренние спецификации, не опубликованные в MSDN

**Проблема:** Многие аспекты NT ядра недокументированы публично. Для реализации совместимости POLER-OS должен определить поведение — но если поведение является коммерческой тайной, его реверс-инжиниринг может быть спорным.

**Однако:**
- *Sega v. Accolade* установил, что реверс-инжиниринг для совместимости — добросовестное использование, даже если информация является коммерческой тайной
- Обратный инжиниринг «чёрного ящика» (наблюдение за поведением) **не является** кражей коммерческой тайны
- Декомпиляция/дизассемблирование для получения интерфейсной информации — также законно по прецеденту

**Реальный риск:** Низкий-средний. POLER-OS должен избегать утечек исходного кода Windows (даже если они есть в интернете).

### 5.5. Что конкретно может сделать Microsoft?

| Тип иска | Вероятность | Основание | Риск для POLER-OS |
|----------|-------------|-----------|-------------------|
| Copyright infringement (копирование кода) | Низкая | POLER-OS не копирует код | Минимальный, если соблюдена чистая комната |
| Patent infringement | Средняя | NTFS, ReFS, BitLocker и др. | Средний, если POLER-OS реализует запатентованные технологии |
| Trademark infringement | Низкая | Использование товарных знаков | Минимальный, при правильном позиционировании |
| Trade secret misappropriation | Низкая | Использование leak-исходников | Минимальный, при запрете на утечки |
| DMCA § 1201 (anti-circumvention) | Низкая | POLER-OS не обходит DRM/TPM | Минимальный |
| Breach of EULA | Низкая | POLER-OS не использует Windows SDK для копирования | Минимальный |

### 5.6. Что Microsoft НЕ может сделать

1. **Подать иск за реализацию API** — после Oracle v. Google это юридически слабая позиция
2. **Запретить совместимость** — прецедентное право защищает право на совместимость
3. **Подать иск за имена функций** — это функциональное использование, не copyrightable
4. **Подать иск за поведение API** — поведение = метод операции = не охраноспособно

## 6. Интерфейс vs. Реализация: Критическое юридическое различие

### 6.1. Спектр охраноспособности

```
Не охраноспособно ←————————————————————————————→ Охраноспособно
   (интерфейс)                                         (реализация)

  Имена API          Заголовки         Псевдокод      Исходный код
  Сигнатуры          SDK .h файлы      Алгоритмы      Бинарный код
  Семантика          Структуры         Оптимизации    Конкретное
  Протоколы          Константы         реализации     выражение
```

### 6.2. Конкретные примеры для POLER-OS

| Элемент | Тип | Охраноспособен? | Действие POLER-OS |
|---------|-----|-----------------|-------------------|
| `NTSTATUS NtCreateFile(PHANDLE, ACCESS_MASK, POBJECT_ATTRIBUTES, ...)` | Интерфейс | ❌ Нет | ✅ Использовать |
| `#define STATUS_SUCCESS ((NTSTATUS)0x00000000L)` | Константа | ❌ Нет | ✅ Использовать |
| Структура `OBJECT_ATTRIBUTES` | Интерфейс | ❌ Нет | ✅ Использовать |
| Код внутри NtCreateFile в ntoskrnl.exe | Реализация | ✅ Да | ❌ Не копировать |
| Алгоритм балансировки handle table | Серая зона | ⚠️ Зависит | Изучить, реализовать самостоятельно |
| Формат on-disk структуры NTFS | Серая зона | ⚠️ Патенты | См. раздел 4.2 |
| Порядок параметров NtCreateFile | Интерфейс | ❌ Нет | ✅ Использовать |
| Возвращаемые значения и коды ошибок | Интерфейс | ❌ Нет | ✅ Использовать |
| Оптимизация I/O priority в IRP handling | Реализация | ✅ Да | ❌ Не копировать |

### 6.3. Практический тест для POLER-OS

При реализации любого компонента задайте вопрос:

> **«Если бы Windows API был полностью другим (другие имена, другая семантика), был бы этот код возможен?»**

Если ответ «нет» — код функционально обусловлен API и, вероятно, не охраноспособен.  
Если ответ «да» — код содержит творческий выбор, который может быть охраноспособен.

## 1. PE Loader

### 1.1. Что должен делать PE Loader

PE Loader — загрузчик Portable Executable файлов — критически важный компонент для Windows-совместимости. Он должен:

1. **Парсинг PE/COFF заголовков:**
   - DOS header (MZ signature)
   - PE signature (PE\0\0)
   - COFF File Header (Machine, NumberOfSections, Characteristics)
   - Optional Header (Magic: PE32 vs PE32+, AddressOfEntryPoint, ImageBase, SizeOfImage)
   - Section Table (VirtualAddress, VirtualSize, SizeOfRawData, Characteristics)

2. **Загрузка секций:**
   - Маппинг секций в виртуальную память (VirtualAlloc-эквивалент)
   - Выравнивание (SectionAlignment, FileAlignment)
   - Обработка характеристик секций (IMAGE_SCN_MEM_READ, IMAGE_SCN_MEM_WRITE, IMAGE_SCN_MEM_EXECUTE)

3. **Обработка релокаций:**
   - Если ImageBase занят → релокация (IMAGE_REL_BASED_DIR64, IMAGE_REL_BASED_ABSOLUTE)
   - Парсинг .reloc секции

4. **Разрешение импортов (IAT — Import Address Table):**
   - Парсинг Import Directory Table
   - Поиск DLL (см. раздел 2 — DLL Loader)
   - Разрешение функций по имени или ординалу
   - Заполнение IAT

5. **Обработка TLS (Thread Local Storage):**
   - TLS Directory → TLS callbacks
   - Вызов TLS callbacks перед entry point

6. **Обработка исключений:**
   - .pdata секция (Runtime Function Table)
   - Регистрация в structured exception handling (SEH) / Vectored Exception Handling (VEH)

7. **Вызов entry point (DllMain / mainCRTStartup)**

### 1.2. Что проверяют античиты

Античиты проверяют PE Loader на:

1. **Целостность загруженного модуля** — совпадает ли in-memory образ с on-disk? Проверяют:
   - Хеш секций .text, .rdata
   - IAT hooks (подмена адресов в IAT)
   - Inline hooks (JMP-инструкции в начале функций)

2. **Корректность релокаций** — все ли релокации применены правильно?

3. **Секции с неожиданными правами** — RWX секции подозрительны

4. **Необычные точки входа** — entry point за пределами .text

5. **Unsigned DLL injection** — загрузка неподписанных DLL в процесс игры

6. **DLL load order** — загружена ли настоящая DLL или подмена?

### 1.3. Сложность и оценка

| Аспект | Сложность | Время (оценка) | Зависимости |
|--------|-----------|----------------|-------------|
| Базовый PE парсинг | Средняя | 2-3 недели | VMM, VFS |
| Релокации | Средняя | 1-2 недели | VMM |
| IAT resolution | Высокая | 3-4 недели | DLL Loader |
| TLS callbacks | Низкая | 3-5 дней | Thread support |
| SEH/VEH | Высокая | 4-6 недель | Exception handling в ядре |
| Manifest/SxS | Высокая | 3-4 недели | XML парсер, WinSxS emulation |

**Итого:** ≈14-23 недели полной реализации PE Loader с поддержкой всех фич.

## 2. DLL Loader

### 2.1. Порядок поиска DLL (DLL Search Order)

Windows использует строгий порядок поиска DLL:

1. Директория, из которой загружен приложение
2. Системная директория (`C:\Windows\System32`)
3. 16-битная системная директория (`C:\Windows\System`) — legacy
4. Директория Windows (`C:\Windows`)
5. Текущая директория
6. Директории в PATH

**С SAFESEARCHALL:**
1. Known DLLs (из `HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Control\Session Manager\KnownDLLs`)
2. Директория приложения
3. Системная директория

### 2.2. Known DLLs

Known DLLs — DLL, которые всегда загружаются из системной директории и разделяют секции между процессами:

```
kernel32.dll, user32.dll, gdi32.dll, advapi32.dll,
ntdll.dll, shell32.dll, comctl32.dll, ole32.dll,
oleaut32.dll, msvcrt.dll, msvcp140.dll, vcruntime140.dll
```

Для POLER-OS: Known DLLs должны быть shim-библиотеками. Каждый процесс получает свою копию (или shared mapping) `poler-ntdll.dll`, `poler-kernel32.dll` и т.д.

### 2.3. Side-by-Side (SxS) Manifests

SxS — механизм Windows для управления версиями DLL:

- Приложение содержит manifest (XML-файл), указывающий требуемые версии DLL
- WinSxS директория содержит несколько версий одной DLL
- Loader выбирает правильную версию по manifest

**Сложность:** Высокая. SxS — одна из самых сложных частей Windows Loader.

**Для POLER-OS:** Минимальная реализация — поддержка assembly identity и redirection. Полная SxS может не потребоваться для большинства приложений.

### 2.4. API Set

API Set — механизм Windows для разделения API между DLL:

- `api-ms-win-core-file-l1-1-0.dll` → перенаправляет к `kernelbase.dll`
- Реально не являются DLL — это таблицы перенаправления

**Для POLER-OS:** Необходимо реализовать API Set resolution. При загрузке DLL с именем `api-ms-win-*` — перенаправлять к соответствующей shim-DLL.

### 2.5. Сложность

| Аспект | Сложность | Время (оценка) |
|--------|-----------|----------------|
| DLL search order | Низкая | 1 неделя |
| Known DLLs | Средняя | 2 недели |
| Delay-load DLL | Средняя | 2 недели |
| SxS manifests | Высокая | 4-6 недель |
| API Set resolution | Средняя | 2-3 недели |
| DLL injection (LoadLibrary) | Средняя | 2 недели |

**Итого:** ≈13-17 недель.

## 3. Object Manager

### 3.1. Обзор архитектуры NT Object Manager

Object Manager (Ob) — это подсистема ядра NT, управляющая всеми объектами:

```
\ (root)
├── ?? (или \DosDevices) → символические ссылки
│   ├── C: → \Device\HarddiskVolume1
│   ├── COM1 → \Device\Serial0
│   └── CON → \Device\ConDrv
├── Device
│   ├── HarddiskVolume1
│   ├── ConDrv
│   ├── Null
│   └── Tcp
├── BaseNamedObjects
│   ├── MySharedMemory (Section)
│   └── MyMutex (Mutex)
├── KernelObjects
│   ├── GlobalEvent
│   └── LowMemoryCondition
├── ObjectTypes
│   ├── Process
│   ├── Thread
│   ├── File
│   ├── Section
│   └── ...
└── Registry
    └── ← точка монтирования реестра
```

### 3.2. Что необходимо реализовать

1. **Иерархическое пространство имён** — строковые пути с `\` как разделителем
2. **Типы объектов** — каждый объект имеет тип (Process, Thread, File, Section, Event, Mutex, Semaphore, Timer, etc.)
3. **Символические ссылки** — `\??\C:` → `\Device\HarddiskVolume1`
4. **Дескрипторы (handles)** — per-process handle table с наследованием
5. **Счётчик ссылок** — автоматическое уничтожение при обнулении
6. **Security descriptors** — ACL на объектах
7. **Waitable objects** — все типы с состоянием signal/non-signal

### 3.3. Сложность и что проверяют античиты

Античиты интенсивно используют Object Manager:

1. **NtOpenDirectoryObject** — перечисление объектов в `\Device`
2. **NtQueryDirectoryObject** — обнаружение скрытых устройств
3. **NtOpenSymbolicLinkObject** — проверка символических ссылок
4. **NtQueryObject** — получение информации об объектах по handle
5. **NtDuplicateObject** — дублирование дескрипторов между процессами

**Сложность:** Очень высокая. Object Manager — фундамент всего NT API. Без него не работают:
- Файловый ввод/вывод (файлы — это объекты)
- IPC (секции, порты — это объекты)
- Синхронизация (events, mutexes — это объекты)
- Безопасность (дескрипторы ссылаются на объекты)

**Оценка времени:** 8-12 недель для полной реализации.

## 4. Registry (Реестр)

### 4.1. Полный формат hive или упрощённый?

Windows Registry хранится в бинарном формате hive. Для POLER-OS есть два подхода:

**Подход A: Полная реализация hive-формата**
- ✅ Можно монтировать настоящие hive-файлы Windows
- ✅ Совместимость с regedit и другими инструментами
- ❌ Сложность: hive-формат недокументирован полностью, содержит много внутренних деталей (hbins, key nodes, value nodes, security descriptors, class names)
- ❌ Риск: hive-формат может содержать запатентованные элементы

**Подход B: Упрощённый реестр (рекомендуемый)**
- ✅ Реализовать только семантику Registry API (RegOpenKeyEx, RegQueryValueEx, RegSetValueEx, RegEnumKeyEx, RegCloseKey)
- ✅ Хранить данные в собственном формате (JSON, SQLite, бинарный)
- ✅ Загружать предопределённые ключи из конфигурационных файлов
- ❌ Нельзя монтировать настоящие hive-файлы

**Рекомендация:** Подход B. Подавляющее большинство приложений использует Registry API, а не прямой доступ к hive-файлам. Античиты тоже используют API.

### 4.2. Какие ключи читают античиты

Критические ключи, которые ДОЛЖНЫ присутствовать:

```
HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\Windows NT\CurrentVersion
  - CurrentVersion = "10.0"
  - CurrentBuild = "19041"
  - ProductName = "Windows 10 Pro"
  - BuildLabEx = "19041.1.amd64fre.vb_release.191206-1406"
  - DigitalProductId (зашифрованный)
  - InstallationType = "Client"
  - RegisteredOwner, RegisteredOrganization

HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Control\Session Manager
  - KnownDLLs (подключи)
  - Environment
  - CDS

HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Control\ComputerName
  - ComputerName

HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\Cryptography
  - MachineGuid (уникальный GUID машины)

HKEY_LOCAL_MACHINE\HARDWARE\DESCRIPTION\System
  - SystemBiosVersion
  - Identifier

HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Control\SystemInformation
  - SystemProductName
  - SystemManufacturer

HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\Windows\CurrentVersion
  - ProgramFilesDir
  - CommonFilesDir
```

**Сложность:** Средняя. Реестр — не самая сложная часть, но требует внимательности к деталям.

**Оценка времени:** 4-6 недель (с предзаполненными ключами).

## 5. Security Reference Monitor (SRM)

### 5.1. Обзор модели безопасности NT

SRM — подсистема, реализующая контроль доступа:

1. **Access Token** — маркер безопасности процесса:
   - User SID
   - Group SIDs
   - Privileges (SeDebugPrivilege, SeLoadDriverPrivilege, etc.)
   - Integrity Level (Low, Medium, High, System)
   - Owner SID, Primary Group SID

2. **Security Descriptor** — дескриптор безопасности объекта:
   - Owner SID
   - Group SID
   - DACL (Discretionary ACL) — кто имеет доступ
   - SACL (System ACL) — аудит доступа

3. **Access Check** — алгоритм проверки доступа:
   - Token vs DACL → granted/denied
   - Integrity level check (Mandatory Integrity Control)
   - Privilege check

4. **Impersonation** — поток может временно принять маркер другого пользователя:
   - SecurityImpersonation, SecurityDelegation уровни
   - Копирование токена (DuplicateTokenEx)

### 5.2. Что нужно для античитов

Античиты используют SRM для:

1. **Проверка привилегий** — SeDebugPrivilege позволяет открыть любой процесс
2. **Проверка integrity level** — игры работают на Medium IL, античит — на System IL
3. **Impersonation** — для доступа к файлам от имени пользователя
4. **Token manipulation detection** — обнаружение изменения токена

### 5.3. Сложность

| Аспект | Сложность | Время (оценка) |
|--------|-----------|----------------|
| Базовый token (SID, Groups) | Средняя | 2-3 недели |
| Privilege check | Низкая | 1 неделя |
| DACL/SACL evaluation | Высокая | 3-4 недели |
| Integrity Levels | Средняя | 1-2 недели |
| Impersonation | Средняя | 2 недели |
| S4U logon (services) | Высокая | 3-4 недели |

**Упрощённая реализация:** Можно начать с «всё разрешено» и постепенно добавлять ограничения. Античиты не требуют полной ACL модели — им достаточно базовых проверок.

**Оценка времени:** 4-6 недель для минимальной рабочей реализации.

## 6. ALPC (Advanced Local Procedure Call)

### 6.1. Что такое ALPC

ALPC — это высокопроизводительный IPC-механизм Windows, используемый для коммуникации между процессами на одной машине. Это эволюция LPC (Local Procedure Call).

**Ключевые характеристики:**
- Двунаправленные соединения (connection port + communication port)
- Три типа сообщений: short (встроено в порт), medium (в буфере), large (через shared section)
- Синхронные и асинхронные операции
- Поддержка передачи дескрипторов между процессами

**API:**
- `NtCreatePort` — создание порта
- `NtConnectPort` — подключение к порту
- `NtListenPort` — ожидание соединения
- `NtAcceptConnectPort` — принятие соединения
- `NtRequestPort` / `NtRequestWaitReplyPort` — отправка сообщения
- `NtReplyPort` / `NtReplyWaitReceivePort` — ответ и ожидание

### 6.2. Используют ли античиты ALPC?

**Да, критически.** ALPC используется:

1. **csrss.exe** — подсистема Win32, общается с ядром через ALPC
2. **LSASS** — служба безопасности (аутентификация, токены)
3. **Service Control Manager** — управление службами
4. **RPC** — Remote Procedure Call использует ALPC как локальный транспорт
5. **Античит сервисы** — общаются с kernel driver через ALPC

### 6.3. Сложность

**Очень высокая.** ALPC — один из самых сложных компонентов NT:

- Сложная машина состояний (connection → communication → closed)
- Передача дескрипторов между процессами
- Shared sections для больших сообщений
- Синхронизация (waiting, completion ports)
- Security context inheritance

**Оценка времени:** 10-16 недель для полной реализации. Минимальная реализация для античитов — 6-8 недель.

## 7. LPC (Local Procedure Call)

### 7.1. Legacy, но ещё нужен?

LPC — предшественник ALPC, использовался в Windows NT 4.0 — Windows XP. Начиная с Vista, LPC реализован поверх ALPC.

**Для POLER-OS:** Не нужно реализовывать LPC отдельно. Достаточно ALPC. Старые приложения, использующие LPC, будут работать через ALPC compatibility layer.

**Оценка:** 0 недель (не нужен).

## 8. Scheduler (Планировщик)

### 8.1. Модель планирования NT

Windows NT использует приоритетную многозадачность:

**Приоритеты потоков:**
- 0: Passive (zero-page thread)
- 1-15: Dynamic priority (variable class)
  - 1-2: Idle
  - 3-4: Below Normal
  - 5-6: Normal (по умолчанию)
  - 7-8: Above Normal
  - 9-10: High
  - 11-15: Real-time range (dynamic)
- 16-31: Real-time priority class

**Квант (quantum):**
- Windows Client: 2 clock ticks ≈ 30ms
- Windows Server: 12 clock ticks ≈ 180ms
- Variable quantum: foreground boost (+1, +2)

**Affinity:**
- Processor affinity (mask — на каких CPU может выполняться)
- NUMA awareness
- Ideal node / ideal processor

**Priority boost:**
- I/O completion boost (+1)
- Wait completion boost
- Foreground boost
- Anti-starvation boost

### 8.2. IRQL implications

IRQL (Interrupt Request Level) — концепция NT, не имеющая прямого аналога в UNIX:

```
PASSIVE_LEVEL (0)    — user-mode, pageable
APC_LEVEL (1)        — APC delivery, pageable
DISPATCH_LEVEL (2)   — DPC dispatch, non-pageable
                     — spinlocks must be held
DEVICE_LEVEL (3-26)  — device-specific
PROFILE_LEVEL (27)   — profiling
CLOCK_LEVEL (28)     — clock interrupt
POWER_LEVEL (29)     — power failure
HIGH_LEVEL (31)      — machine check, non-maskable
```

POLER-OS не обязан реализовывать IRQL в точности — это внутренняя деталь NT ядра. Однако:
- Концепция приоритетов прерываний — общая для всех ОС
- POLER-OS может использовать собственную модель (например, стандартные x86 interrupt priorities)

### 8.3. Что проверяют античиты

1. **Thread priority** — античиты устанавливают высокий приоритет и проверяют, что он соблюдается
2. **Affinity** — некоторые античиты привязывают потоки к конкретным CPU
3. **Thread creation timing** — обнаружение необычных паттернов создания потоков
4. **Quantum** — косвенные проверки через timing

**Сложность:** Средняя. POLER-OS уже имеет планировщик, нужно расширить поддержкой приоритетов NT.

**Оценка времени:** 3-4 недели для адаптации.

## 9. Memory Manager (Менеджер памяти)

### 9.1. VAD Tree (Virtual Address Descriptor)

VAD — структура, описывающая регион виртуальной памяти процесса:

```
VAD Entry:
  - StartingVpn (Virtual Page Number)
  - EndingVpn
  - CommitCharge
  - VadType (Section, Image, Mapped, etc.)
  - Protection (READ, WRITE, EXECUTE, COPY_ON_WRITE)
  - LeftChild, RightChild (AVL tree)
  - ControlArea (для section objects)
```

VAD организован в AVL-дерево для быстрого поиска региона по адресу.

**Для POLER-OS:** Нужно реализовать эквивалент VAD с теми же свойствами:
- AVL или red-black дерево регионов
- Protection bits (RWX, COPY_ON_WRITE, GUARD)
- Section backing (memory-mapped files)

### 9.2. Section Objects

Section (секция) — это объект, представляющий shared memory или memory-mapped file:

- `NtCreateSection` → создаёт секцию (file-backed или pagefile-backed)
- `NtMapViewOfSection` → маппит секцию в адресное пространство
- Shared sections — основа для shared memory между процессами
- Copy-on-write — для fork-подобного поведения

**Критически важно для античитов:** Shared memory используется для коммуникации между anti-cheat service и anti-cheat driver.

### 9.3. Page Fault Handling

NT обрабатывает page faults через диспетчеризацию:

1. **Valid PTE** → normal access
2. **PTE = 0** → demand-zero fault (allocate zeroed page)
3. **PTE = Transition** → в списках standby/modified → реактивация
4. **PTE = Prototype PTE** → section-backed → read from file
5. **PTE = Demand Zero** → allocate + zero
6. **Guard page** → STATUS_GUARD_PAGE_VIOLATION → stack expansion

### 9.4. Сложность

| Аспект | Сложность | Время (оценка) |
|--------|-----------|----------------|
| VAD tree (AVL) | Высокая | 3-4 недели |
| Section objects | Очень высокая | 6-8 недель |
| Page fault handling | Высокая | 4-6 недель |
| Shared memory | Высокая | 3-4 недели |
| Copy-on-write | Средняя | 2-3 недели |
| Working set management | Высокая | 4-6 недель |
| Page file (swap) | Высокая | 4-6 недель |

**Итого:** ≈26-37 недель. Это самый объёмный компонент.

## 10. Driver Model

### 10.1. Можно ли загружать .sys драйверы?

**Краткий ответ:** Нет, и не нужно пытаться.

**Развернутый ответ:**

Windows kernel drivers (.sys) — это PE DLL, работающие в Ring 0. Они:
- Вызывают内核-функции (ntoskrnl.exe exports)
- Используют IRQL модели
- Работают с устройствами через IRP (I/O Request Packets)
- Зависят от конкретных структур ядра (DRIVER_OBJECT, DEVICE_OBJECT, IRP)

Для загрузки .sys драйвера напрямую POLER-OS должен был бы:
1. Иметь бинарно-совместимый ntoskrnl.exe с теми же экспортами
2. Иметь те же смещения в структурах (DRIVER_OBJECT, DEVICE_OBJECT)
3. Иметь тот же IRP flow
4. Иметь ту же модель IRQL

**Это практически невозможно без клонирования всего NT kernel.**

### 10.2. Рекомендуемый подход

POLER-OS должен использовать **собственный драйверный формат** (как и планируется — микроядро с собственным ABI). Для Windows-совместимости:

1. **Пользовательские драйверы** — shim-слой, транслирующий Win32 device I/O в native POLER-OS syscalls
2. **Anti-cheat драйверы** — см. ЧАСТЬ 3 (это отдельная сложная тема)
3. **Файловая система** — POLER-FS через собственный VFS
4. **GPU** — Vulkan напрямую, DirectX через DXVK

### 10.3. IRP (I/O Request Packet) — нужно ли?

IRP — фундаментальная концепция Windows I/O:

```
Irp {
  Type: READ/WRITE/DEVICE_CONTROL/INTERNAL_DEVICE_CONTROL
  Flags: SYNCHRONOUS_API, etc.
  IoStatus: Status + Information
  UserBuffer, SystemBuffer, MdlAddress
  StackLocation[]: {
    MajorFunction, MinorFunction
    Parameters (read/write length, ioctl code)
    DeviceObject, FileObject
  }
}
```

Для POLER-OS: IRP не нужен в ядре. Shim-слой может транслировать Win32 I/O API в POLER-OS native I/O.

**Оценка:** 0 недель на IRP в ядре. Shim-трансляция — 2-3 недели.

## 11. System Calls

### 11.1. Проблема нестабильных номеров syscall

Как указано выше, номера syscall меняются между версиями Windows. Античиты знают это и не полагаются на конкретные номера.

**Стратегия POLER-OS:**

1. **Не использовать номера syscall Windows.** POLER-OS имеет собственный syscall interface.
2. **NT API реализуется в user-mode shim (poler-ntdll.dll).** Shim предоставляет функции NtCreateFile, NtReadFile и т.д., которые внутри вызывают native POLER-OS syscalls.
3. **Прямой syscall из приложений** — если приложение делает `syscall` напрямую с номером NtCreateFile:
   - Per-process syscall table перехватывает вызов
   - Upcall в shim-библиотеку для трансляции
   - 2 ring switches — дорого, но редко встречается

### 11.2. NtDll как точка входа

`ntdll.dll` — это user-mode DLL, которая:
- Экспортирует все NT native API (NtCreateFile, NtReadFile, ...)
- Делает `syscall` инструкцию для перехода в ядро
- Содержит user-mode части RTL (RtlAllocateHeap, RtlCreateUnicodeString, etc.)

Для POLER-OS: `poler-ntdll.dll` заменяет ntdll.dll:
- Экспортирует те же функции (имена и сигнатуры)
- Внутри вызывает POLER-OS native syscalls
- Реализует RTL функции (heap, string, security)

**Сложность:** Очень высокая. ntdll.dll экспортирует ≈2000+ функций.

**Оценка времени:** 20-30 недель для полной реализации (но можно начать с подмножества).

## 12. User Mode

### 12.1. ntdll.dll

См. раздел 11.2. Дополнительно:

- **LdrInitializeThunk** — точка входа для инициализации процесса
- **KiUserExceptionDispatcher** — user-mode диспетчер исключений
- **KiUserApcDispatcher** — user-mode APC delivery
- **KiUserCallbackDispatcher** — callback из ядра в user mode
- **Rtl* functions** — ≈1500+ runtime library функций

### 12.2. csrss.exe (Client/Server Runtime Subsystem)

csrss.exe — критический процесс Windows, реализующий:
- Win32 console (консольный ввод/вывод)
- Hard error handling (MessageBox для крашей)
- Создание/завершение процессов и потоков (уведомления)
- Драйвер консоли (ConDrv)

**Для POLER-OS:** Необходимо реализовать эквивалент csrss.exe:
- Console I/O через POLER-OS native IPC
- Process/thread notification mechanism
- Hard error → log/notify вместо MessageBox

**Сложность:** Высокая. csrss.exe — большой компонент.

**Оценка времени:** 8-12 недель.

### 12.3. Subsystem Architecture

Windows NT изначально проектировался как мульти-подсистемная ОС:
- Native NT subsystem (ntdll.dll)
- Win32 subsystem (kernel32.dll, user32.dll, gdi32.dll)
- POSIX subsystem (устарел)
- OS/2 subsystem (устарел)

**Для POLER-OS:** Реализовать только Win32 subsystem. Это соответствует текущему плану shim-архитектуры.

## 13. Kernel Mode

### 13.1. Какие структуры ядра должны быть точными?

**Критический вопрос:** Приложения НЕ имеют прямого доступа к kernel structures. Однако kernel-mode DRIVERS имеют. Если POLER-OS не загружает .sys драйверы (см. раздел 10), точная совместимость структур не требуется.

**Структуры, которые должны быть точными (для user-mode совместимости):**

| Структура | Точность | Почему |
|-----------|----------|--------|
| TEB (Thread Environment Block) | Высокая | User-mode, доступен через GS:[0x30] в x64 |
| PEB (Process Environment Block) | Высокая | User-mode, доступен через TEB→PEB |
| PEB_LDR_DATA | Высокая | Список загруженных модулей (PEB→Ldr) |
| LDR_DATA_TABLE_ENTRY | Высокая | Информация о загруженном модуле |
| NT_TIB | Высокая | Thread Information Block (top стека, etc.) |
| CLIENT_ID | Средняя | PID/TID, используется в API |
| OBJECT_ATTRIBUTES | Высокая | Параметр для NtOpen* функций |
| IO_STATUS_BLOCK | Высокая | Результат NtReadFile/NtWriteFile |
| UNICODE_STRING | Высокая | Формат строк в NT API |
| SECURITY_DESCRIPTOR | Средняя | Для ACL operations |
| EPROCESS | Низкая | Не видна из user mode |
| ETHREAD | Низкая | Не видна из user mode |
| KTHREAD | Низкая | Не видна из user mode |
| DRIVER_OBJECT | Низкая | Только kernel mode |

### 13.2. TEB — критически важная структура

```
TEB (x64, offset от GS:0):
  0x000: SEH chain
  0x008: Stack limit (DEPRECATED в x64)
  0x010: SubSystemTib
  0x018: FiberData
  0x020: ArbitraryUserPointer
  0x028: TEB address (self-pointer)
  0x030: Process Environment Block (PEB pointer)
  0x038: LastErrorValue
  0x040: Count of owned critical sections
  0x048: CSR Client Thread
  0x050: Win32 Thread Info
  ...
  0x0F8: GDI TEB Batch
  0x100: Real Process ID
  0x108: Real Thread ID
  ...
  0x14C0: deallocation stack
  ...
  0x1748: WOW32Reserved
  ...
  0x17F8: FlsData
  ...
```

**POLER-OS ДОЛЖЕН** предоставлять TEB по GS:[0x30] (x64) или FS:[0x18] (x86) с корректными смещениями для:
- Self-pointer (offset 0x28 для x64)
- PEB pointer (offset 0x30)
- LastErrorValue (offset 0x38)
- PID/TID (offset 0x100, 0x108)
- Stack limits
- TLS slots

### 13.3. PEB — вторая критическая структура

```
PEB (x64):
  0x000: InheritedAddressSpace
  0x001: ReadImageFileExecOptions
  0x002: BeingDebugged  ← АНТИЧИТЫ ПРОВЕРЯЮТ!
  0x003: BitField
  0x008: Mutant
  0x010: ImageBaseAddress
  0x018: Ldr (PEB_LDR_DATA pointer) ← список модулей
  0x020: ProcessParameters (RTL_USER_PROCESS_PARAMETERS)
  0x028: SubSystemData
  0x030: ProcessHeap
  0x038: FastPebLock
  0x040: AtlThunkSListPtr
  0x048: IFEOKey
  0x050: CrossProcessFlags
  ...
  0x0BC: NumberOfHeaps
  0x0C0: MaximumNumberOfHeaps
  0x0C8: ProcessHeaps
  ...
  0x118: OSMajorVersion   ← АНТИЧИТЫ ЧИТАЮТ!
  0x11C: OSMinorVersion
  0x120: OSBuildNumber
  0x124: OSPlatformId
  ...
```

**Критические поля для античитов:**
- `BeingDebugged` (0x002) — если TRUE, античит может отказать
- `NtGlobalFlag` (0x068 для x86, 0x0BC для x64) — отладочные флаги
- `Ldr` → список загруженных модулей — античиты проверяют на инъекции
- `OSMajorVersion`, `OSMinorVersion`, `OSBuildNumber` — версия ОС
- `ProcessHeap` — античиты проверяют heap на модификации

### 13.4. Что МОЖЕТ отличаться

- Внутренние структуры ядра (EPROCESS, ETHREAD, KTHREAD) — приложения их не видят
- Алгоритмы планирования — достаточно семантической совместимости
- Формат page tables — POLER-OS использует свой VMM
- Драйверная модель — POLER-OS использует свою

# ЧАСТЬ 3: СОВМЕСТИМОСТЬ С АНТИЧИТАМИ

---

## Введение: Разделение трёх категорий

Крайне важно чётко разделять:

1. **Технически достижимо через корректную реализацию API** — POLER-OS может это сделать
2. **Зависит от политики разработчика античита** — POLER-OS технически может, но античит отказывает по политическим причинам
3. **Фундаментально невозможно без сотрудничества** — POLER-OS не может это сделать независимо

---

## 1. Easy Anti-Cheat (EAC) — Epic Games

### 1.1. Архитектура EAC

EAC состоит из:
- **EAC kernel driver** (`EasyAntiCheat.sys`) — Ring 0, загружается через Windows Driver Framework
- **EAC user-mode module** — внедряется в игровой процесс
- **EAC service** — фоновая служба, общается с сервером Epic
- **EAC launcher** — проверяет целостность перед запуском

### 1.2. Что проверяет EAC

**Kernel-mode (driver):**
1. **Идентификация ОС** — проверяет версию Windows через `RtlGetVersion` и `NtQuerySystemInformation`
2. **Обнаружение неподписанных драйверов** — перечисляет `\Driver` и `\FileSystem` через Object Manager
3. **Проверка SSDT (System Service Dispatch Table)** — обнаруживает hooking системных вызовов
4. **Проверка IDT/GDT** — обнаруживает модификации interrupt descriptor table
5. **Мониторинг создания процессов** — `PsSetCreateProcessNotifyRoutine`
6. **Проверка целостности NtDll** — сравнение in-memory с on-disk
7. **Обнаружение DLL injection** — перечисление модулей через PEB→Ldr
8. **Проверка открытых дескрипторов** — `NtQuerySystemInformation(SystemHandleInformation)`
9. **Обнаружение отладчиков** — `NtQueryInformationProcess(ProcessDebugPort)`

**User-mode (module):**
1. **Проверка PEB.BeingDebugged** — обнаружение отладчика
2. **Проверка NtGlobalFlag** — отладочные флаги
3. **Сканирование памяти процесса** — обнаружение модификаций кода
4. **Проверка целостности IAT** — Import Address Table не должна быть модифицирована
5. **Heartbeat** — регулярные проверки с сервером

### 1.3. Анализ по трём категориям

**✅ Технически достижимо через корректную реализацию:**

| Функция | Как реализовать | Сложность |
|---------|----------------|-----------|
| Идентификация ОС (версия) | Правильные значения в PEB и RtlGetVersion | Низкая |
| Перечисление драйверов | Object Manager: `\Driver`, `\FileSystem` | Средняя |
| Обнаружение отладчиков | PEB.BeingDebugged = FALSE, NtQueryInformationProcess возвращает STATUS_PORT_NOT_SET | Низкая |
| Проверка целостности NtDll | poler-ntdll.dll возвращает корректные данные при сравнении | Средняя |
| Обнаружение DLL injection | Корректный PEB→Ldr, только легитимные модули | Средняя |
| Проверка открытых дескрипторов | NtQuerySystemInformation с корректными данными | Высокая |
| Проверка IAT | Shim-загрузчик не модифицирует IAT после загрузки | Низкая |
| Heartbeat | Сетевой стек + корректная TLS-коммуникация | Средняя |
| SSDT проверки | POLER-OS не хукает SSDT (его нет) — нужно, чтобы EAC driver не видел SSDT hooking | N/A (см. ниже) |

**⚠️ Зависит от политики EAC:**

| Проблема | Пояснение |
|----------|-----------|
| **Kernel driver signing** | EAC driver должен быть подписан Microsoft EV сертификатом и пройти WHQL. POLER-OS НЕ может подписать драйвер через Microsoft — POLER-OS не Windows. Это **политическое ограничение**, не техническое. |
| **Проверка подписи ОС** | EAC может проверять, что ядро подписано Microsoft. POLER-OS ядро подписано собственным ключом. |
| **Отсутствие в whitelist** | EAC может поддерживать whitelist ОС. Если POLER-OS не в whitelist — отказ. |
| **Версия Windows** | EAC может требовать конкретную версию Windows (например, «Windows 10 1903+»). POLER-OS может сообщить эту версию, но EAC может проверять глубже. |

**❌ Фундаментально невозможно без сотрудничества:**

| Проблема | Пояснение |
|----------|-----------|
| **Загрузка EAC kernel driver** | EAC driver (.sys) скомпилирован для Windows kernel. Он вызывает ntoskrnl.exe экспорты. POLER-OS не имеет ntoskrnl.exe. Невозможно загрузить без полного клонирования kernel interface. |
| **WHQL подпись** | POLER-OS не может получить WHQL подпись от Microsoft — это требует Windows Hardware Developer Center registration и прохождения тестов на настоящей Windows. |
| **Kernel-mode anti-tamper** | EAC driver выполняет проверки в Ring 0, которые предполагают конкретную архитектуру Windows kernel. POLER-OS не может эмулировать это без полного клонирования. |

### 1.4. Вывод по EAC

**Без сотрудничества с Epic Games** — полная совместимость с EAC невозможна. POLER-OS может реализовать все user-mode проверки корректно, но EAC kernel driver не загрузится на POLER-OS.

**Возможный путь:** Epic Games могла бы выпустить **POLER-OS-совместимую версию EAC** — user-mode only, без kernel driver. Но это требует **политического решения** Epic.

## 2. BattlEye

### 2.1. Архитектура BattlEye

BattlEye аналогичен EAC:
- **BE kernel driver** (`BEDaisy.sys`) — Ring 0
- **BE module** — внедряется в игровой процесс
- **BE service** — фоновая служба

### 2.2. Специфические особенности BattlEye

1. **Более агрессивное сканирование памяти** — BattlEye сканирует больше регионов памяти
2. **Проверка кодовых секций** — сравнение с on-disk образами
3. **Обнаружение manual mapping** — DLL, загруженные не через стандартный loader
4. **Проверка потоков** — обнаружение скрытых/приостановленных потоков
5. **Network analysis** — проверка сетевых соединений
6. **Driver enumeration** — через `NtEnumerateDriver` и `MmGetSystemRoutineAddress`

### 2.3. Анализ по трём категориям

Аналогичен EAC. BattlEye имеет те же фундаментальные ограничения:
- Kernel driver не загрузится на POLER-OS
- Требуется сотрудничество с BattlEye GmbH

**Специфическая проблема BattlEye:** BattlEye особенно активно использует kernel-mode проверки. Даже если user-mode компонент работает корректно, отсутствие kernel driver будет замечено.

## 3. Vanguard (Riot Games)

### 3.1. Архитектура Vanguard

Vanguard — самый агрессивный из современных античитов:

- **vgk.sys** — kernel driver, загружается при старте системы (boot-start driver)
- **vgc.sys** — communication driver
- **Riot Client** — user-mode компонент
- **Глубокая интеграция с Windows Security** — использует Hypervisor-protected Code Integrity (HVCI), Credential Guard

### 3.2. Уникальные особенности Vanguard

1. **Boot-start driver** — vgk.sys загружается ДО входа пользователя в систему
2. **HVCI dependency** — требует Virtualization-based Security (VBS) и Hypervisor-protected Code Integrity
3. **UMPK (User Mode Process Kick)** — может завершить любой процесс
4. **EC (Execution Control)** — мониторинг выполнения кода в real-time
5. **Проверка EFI/UEFI** — проверяет, что загрузчик не модифицирован
6. **TPM dependency** — может использовать TPM для attestation

### 3.3. Анализ по трём категориям

**✅ Технически достижимое (user-mode):**
- Корректные ответы на все user-mode API запросы
- Правильные структуры PEB/TEB
- Корректная реализация NtQuerySystemInformation

**⚠️ Политические:**
- Vanguard требует **HVCI** — это технология Windows, зависящая от Hyper-V. POLER-OS не может реализовать HVCI в том же виде.
- Vanguard требует **TPM attestation** — POLER-OS может реализовать TPM поддержку, но attestation будет другим.
- Riot Games **явно не поддерживает** не-Windows платформы — даже Linux с Wine официально не работает с Vanguard.

**❌ Фундаментально невозможное:**
- **vgk.sys boot-start driver** — не загрузится на POLER-OS
- **HVCI/VBS** — требует Hyper-V, POLER-OS не может эмулировать это без сотрудничества Microsoft
- **TPM attestation с Microsoft PKI** — TPM сертификация идёт через Microsoft, POLER-OS не может получить Microsoft-сертифицированный TPM attestation

### 3.4. Вывод по Vanguard

Vanguard — **наименее совместимый** с POLER-OS античит. Его архитектура глубоко интегрирована с Windows security stack (VBS, HVCI, TPM). Даже при сотрудничестве с Riot Games, портирование Vanguard на POLER-OS потребует значительной переработки.

## 4. EQU8

### 4.1. Архитектура EQU8

EQU8 — более «лёгкий» античит:
- **User-mode primarily** — основная работа в user-mode
- **Machine learning** — анализ поведенческих паттернов
- **Server-side validation** — значительная часть проверок на сервере
- **Минимальный kernel component** — если есть, то минимальный

### 4.2. Анализ по трём категориям

**✅ Технически достижимое:**
- Все user-mode проверки — корректная реализация NT API
- Behavioral analysis — если POLER-OS ведёт себя как Windows, ML-модели не заметят разницы
- Server-side validation — сетевой стек + корректные ответы

**⚠️ Политические:**
- EQU8 может проверять версию ОС и отказывать на неизвестных платформах
- EQU8 может требовать конкретные DLL версии

**❌ Фундаментально невозможное:**
- Если EQU8 имеет kernel component (даже минимальный) — та же проблема с .sys драйверами

### 4.3. Вывод по EQU8

EQU8 — **наиболее совместимый** с POLER-OS античит из рассмотренных. Если его kernel component минимален или отсутствует, POLER-OS может обеспечить совместимость через корректную реализацию NT API.

## 6. Возможные стратегии для античит-совместимости

### 6.1. Стратегия A: Сотрудничество с разработчиками античитов

**Подход:** Убедить разработчиков античитов выпустить POLER-OS-совместимую версию.

**Требования:**
1. POLER-OS должен продемонстрировать достаточную рыночную долю
2. POLER-OS должен предоставить стабильный kernel API для античитов
3. POLER-OS должен обеспечить механизм подписи и верификации кода

**Вероятность:** Низкая в краткосрочной перспективе. Но если POLER-OS наберёт популярность — возможно.

### 6.2. Стратегия B: User-mode-only античит

**Подход:** Работать только с играми, использующими user-mode античиты.

**Примеры:**
- VAC (Valve Anti-Cheat) — user-mode
- EQU8 — преимущественно user-mode
- Собственные античиты некоторых игр

**Вероятность:** Средняя. Многие онлайн-игры используют kernel-mode античиты, но не все.

### 6.3. Стратегия C: Собственный античит-фреймворк

**Подход:** POLER-OS предоставляет собственный kernel API для античитов, и разработчики портируют свои античиты.

**Преимущества:**
- POLER-OS контролирует security model
- Античиты работают нативно, без совместимости с Windows
- Чистая архитектура без legacy

**Требования:**
1. Стабильный kernel API для античитов
2. Механизм подписи и верификации
3. Документация и SDK
4. Рыночная доля, мотивирующая разработчиков

### 6.4. Стратегия D: Hypervisor-based подход

**Подход:** POLER-OS работает как гипервизор, а Windows — как guest OS. Античиты работают в настоящей Windows.

**Преимущества:**
- 100% совместимость с античитами
- Не нужно ничего реализовывать — античит работает в настоящей Windows

**Недостатки:**
- Требуется лицензия Windows
- POLER-OS перестаёт быть «независимой ОС» и становится гипервизором
- Производительность (виртуализация overhead)

## 7. Критический вывод по античитам

**Невозможно обеспечить совместимость с kernel-mode античитами (EAC, BattlEye, Vanguard) исключительно через корректную реализацию NT API.**

Причина: эти античиты **загружают kernel drivers** (.sys файлы), которые:
1. Вызывают ntoskrnl.exe экспорты, которых нет в POLER-OS
2. Используют Windows-специфичные структуры (DRIVER_OBJECT, DEVICE_OBJECT, IRP)
3. Требуют WHQL подписи от Microsoft
4. Могут зависеть от VBS/HVCI/TPM

**Это не техническое ограничение POLER-OS — это архитектурное ограничение.** Kernel-mode античиты по определению работают внутри Windows kernel. Они не могут работать в другом ядре без портирования.

**Корректная реализация NT API решает:**
- ✅ Совместимость с user-mode приложениями (включая игры)
- ✅ Совместимость с user-mode античитами
- ✅ Корректное поведение при проверках из user-mode

**Корректная реализация NT API НЕ решает:**
- ❌ Загрузку Windows kernel drivers
- ❌ WHQL подпись
- ❌ VBS/HVCI зависимости
- ❌ TPM attestation через Microsoft PKI

# РЕКОМЕНДАЦИИ

## Юридические

1. **Использовать модифицированную чистую комнату** — даже для одного разработчика
2. **Не дизассемблировать** код Microsoft — только публичная документация и чёрный ящик
3. **Не реализовывать NTFS** — использовать POLER-FS или ext2/4
4. **Не использовать leak-исходники** — абсолютно, без исключений
5. **Документировать процесс** — git-коммиты с пометками [SPEC]/[IMPL]
6. **Не использовать товарные знаки Microsoft** в маркетинге
7. **Рассмотреть OIN участие** для патентной защиты

## Архитектурные

1. **Начать с TEB/PEB** — фундамент для всего остального
2. **PE Loader + DLL Loader** — без этого ничего не запустится
3. **Object Manager** — без него не работают файлы, IPC, синхронизация
4. **Memory Manager (VAD + Sections)** — без этого не работает shared memory
5. **ntdll.dll shim** — основная точка входа для NT API
6. **Не пытаться загружать .sys драйверы** — это тупик
7. **Реализовать собственный античит API** — долгосрочная стратегия

## Античит-совместимость

1. **Приоритировать user-mode античиты** — VAC, EQU8
2. **Сотрудничать с разработчиками античитов** — long-term стратегия
3. **Предоставить POLER-OS Anti-Cheat SDK** — если POLER-OS наберёт популярность
4. **Не обещать kernel-mode античит-совместимость** — это невозможно без сотрудничества
5. **Честно позиционировать** — «совместимость с Win32 приложениями», а не «запускает Valorant»
