Хороший вопрос, но тут есть важное заблуждение которое нужно развеять.

Вулкан и DX12 — не враги
На Windows можно установить и Vulkan и DX12 одновременно. Это не взаимоисключающие вещи. Каждый游戏 выбирает какой API использовать при запуске. Проблема не в установке — проблема в качестве реализации.

Вот как это реально работает:

text

┌─────────────────────────────────────────────┐
│  Игра (Гэншин)                              │
│  Выбирает: Vulkan ИЛИ DX12 при запуске      │
├──────────────┬──────────────────────────────┤
│   DX12 path  │   Vulkan path                │
│   (native)   │   (native)                   │
├──────────────┴──────────────────────────────┤
│  GPU драйвер (NVIDIA/AMD)                   │
│  Реализует ОБА API поверх железа            │
└─────────────────────────────────────────────┘
GPU драйвер — и NVIDIA и AMD — реализует оба API. Железо одно, API два, драйвер переводит оба в одинаковые команды GPU.

Почему «вулкан хуже» или «DX12 хуже»
Это не потому что API плохой, а потому что:

Разные команды разработчиков — одни люди пишут DX12 рендерер, другие Vulkan. Качество разное
Драйверная оптимизация — NVIDIA традиционно лучше оптимизирует DX12 чем Vulkan
Гэншин изначально DX11 — Vulkan рендерер добавили позже, и он сырой
AMD наоборот — у AMD Vulkan часто работает лучше чем DX12
Что это значит для POLER-OS
Тут отличная новость — тебе НЕ нужно реализовывать оба API. Правильная стратегия:

Но есть реальная проблема
Ты прав — проблема есть, но она другая:

Проблема не «Vulkan vs DX12», а «translation overhead + anti-cheat».

Серьёзность
Решение

Anti-cheat — вот настоящий враг, а не Vulkan vs DX12. miHoYo использует kernel-level anti-cheat который не работает даже в Wine/Proton. Для этого нужен либо:

Свой kernel-level драйвер который anti-cheat примет
Договорённость с miHoYo (хаха)
Обход anti-cheat (легально серая зона)
Итого
Тебе нужно реализовать один GPU API — Vulkan. А DX11/DX12 приезжают бесплатно через DXVK/VKD3D. Это не теория — это проверенный путь который Valve использует в Steam Deck.

Проблема «некоторые функции работают хуже» — это проблема конкретной реализации рендерера в игре, а не фундаментальная проблема API. На POLER-OS через shim можно будет точечно патчить проблемные вызовы.

FAT32 (1996)

├── ✓ Журналирование → защита от сбоёв

├── ✓ Большие файлы → до 16 ТБ
├── ✗ Нет встроенного шифрования → нужен LUKS

├── ✗ Не оптимизирована под SSD/NVMe
└── ✗ POSIX ACL — не NT-модель (проблема для совместимости)

Btrfs (2009) / ZFS (2005)
├── ✓ Copy-on-Write

Реализация:
├── Каждый inode содержит Encryption Key ID
├── Ключи хранятся в kernel keyring
├── Алгоритм: AES-256-XTS (как LUKS2, но встроен)
├── Имя файла тоже шифруется (AES-256-CTS)
└── Метаданные (размер, время) — не шифруются для скорости
Кит 2: NT-совместимая модель ACL (не POSIX!)
Это критично для Windows-совместимости:

text

POSIX (Linux):                    NT ACL (Windows / POLER-FS):
─────────────                     ────────────────────────────
rwxrwxrwx                         ACE (Access Control Entry):
├── Owner: rwx                     ├── SID (Security ID)
├── Group: rwx                     ├── Access Mask:
├── Other: rwx                     │   ├── Generic Read
└── Всё. Три уровня.              │   ├── Generic Write
                                   │   ├── Generic Execute
                                   │   ├── Delete
                                   │   ├── Read Permissions
                                   │   ├── Change Permissions
                                   │   ├── Take Ownership
                                   │   └── Synchronize
                                   ├── Type: Allow / Deny
                                   └── Flags: Inherit / NoPropagate

Разница:
POSIX: "Файл открыт для группы" → ГРУБАЯ модель
NT ACL: "Пользователь X может читать, но не удалять,
         а группа Y может всё кроме смены владельца"
         → ТОНИКАЯ модель
Именно поэтому Wine не может полноценно работать с NTFS — POSIX ACL не выражает NT ACL. POLER-FS решает это нативно.

Кит 3: NVMe-first дизайн
text

Традиционные ФС:                 POLER-FS:
────────────────                  ────────
Оптимизация под HDD              Оптимизация под NVMe
├── Минимизировать seek           ├── Параллельные I/O queues
├── Дефрагментация               ├── Multiple submission queues
├── Rotational оптимизации       ├── Zero-copy I/O
└── Один I/O queue               └── NOPs для alignment

POLER-FS NVMe оптимизации:
├── Block size = 4КБ (совпадает с NVMe page)
├── Multi-queue: каждая CPU core → свой I/O queue
│   CPU 0 → SQ 0 → CQ 0
│   CPU 1 → SQ 1 → CQ 1
│   CPU N → SQ N → CQ N
├── IO_uring-подобный интерфейс в ядре
│   ├── Submission ring (user → kernel)
│   ├── Completion ring (kernel → user)
│   └── Zero syscalls для I/O!
├── TRIM/DISCARD встроен в фоновой сборщик
└── Write amplification awareness
    ├── Не перезаписываем мелкие блоки
    └── Copy-on-Write для метаданных
Inode структура POLER-FS
text

struct poler_inode {
    // Базовые поля (как ext4)
    uint32_t    i_mode;         // Файл/директория/симлинк
    uint32_t    i_uid;          // Owner SID hash
    uint32_t    i_gid;          // Group SID hash
    uint64_t    i_size;         // Размер файла (до 16 ЭБ!)
    uint64_t    i_blocks;       // Количество 4КБ блоков
    
    // Временные метки (наносекундная точность)
    uint64_t    i_atime_ns;     // Last access
    uint64_t    i_mtime_ns;     // Last modify
    uint64_t    i_ctime_ns;     // Status change
    uint64_t    i_btime_ns;     // Birth/creation time ← NT требует!
    
    // Указатели на данные
    uint32_t    i_block[12];    // Прямые блоки (48 КБ)
    uint32_t    i_indirect;     // Одинарная косвенность
    uint32_t    i_dindirect;    // Двойная косвенность
    uint32_t    i_tindirect;    // Тройная косвенность
    
    // === POLER-FS уникальные поля ===
    
    // Криптография
    uint8_t     i_enc_algo;     // 0=нет, 1=AES-256-XTS, 2=ChaCha20
    uint8_t     i_key_id;       // ID ключа в kernel keyring
    uint8_t     i_nonce[12];    // Per-inode nonce
    
    // NT ACL
    uint32_t    i_acl_off;      // Смещение ACE-массива
    uint32_t    i_acl_len;      // Количество ACE
    uint32_t    i_sacl_off;     // System ACL (аудит)
    uint32_t    i_sacl_len;     // Количество S-ACE
    
    // Целостность
    uint8_t     i_hash_algo;    // 0=нет, 1=SHA-256, 2=BLAKE3
    uint8_t     i_integrity[32];// SHA-256/BLAKE3 хэш содержимого
    
    // NVMe оптимизации
    uint16_t    i_io_prio;      // I/O priority class
    uint8_t     i_align;        // Alignment requirement (0=4K)
    uint8_t     i_cow_flag;     // Copy-on-Write для этого файла
    
    // Расширенные атрибуты (xattr)
    uint32_t    i_xattr_off;    // Смещение xattr блока
    uint32_t    i_xattr_len;    // Размер xattr
};
// Размер: ~256 байт → 16 inode на 4КБ блок

Фаза 1 (сейчас): VFS абстракция
├── VFS интерфейс: open(), read(), write(), close()

Фаза 2: POLER-FS базовая

├── Журналирование (WAL)
└── mkfs.poler — форматирование

Фаза 3: POLER-FS продвинутая
├── NT ACL реализация (ACE/SID)
├── Kernel keyring + per-inode шифрование
├── Integrity хэши (BLAKE3 per-block)
├── NVMe multi-queue I/O
├── Copy-on-Write для метаданных
└── xattr поддержка

Фаза 4: POLER-FS полная
├── Снапшоты (на основе CoW)
├── Deduplication
├── Transparent compression (LZ4/Zstd)
├── Online defragmentation (для HDD fallback)
├── fsck.poler — проверка и восстановление
└── Migration tool: ext2/NTFS → POLER-FS
Но стартуем с ext2 — и это правильно
Не нужно писать POLER-FS прямо сейчас. Порядок:

text

Сейчас:  VFS → ext2 → работающая ОС с файлами
Потом:   POLER-FS → замена ext2 на родную ФС

Почему:
1. ext2 — простая, хорошо документированная, 20 минут на драйвер
2. Нам нужна РАБОТАЮЩАЯ ОС раньше чем идеальная ФС
3. VFS абстракция позволяет поменять ФС без переписывания верхних слоёв
4. POLER-FS напишем когда ядро уже работает и есть пользователи
Это как дом — сначала фундамент (ядро + VFS), потом стены (ext2), потом отделка (POLER-FS). Но архитектура POLER-FS уже спроектирована — когда дойдёт время, мы точно знаем что строить.
